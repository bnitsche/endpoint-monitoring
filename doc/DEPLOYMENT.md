# Deployment Guide — Endpoint Monitoring

Deployment targets Windows Server with IIS and the .NET 10 Hosting Bundle.  
Two components are deployed independently:

| Component | Project | Hosting |
|---|---|---|
| Web UI | `EndpointMonitoring.Web` | IIS (via ASP.NET Core Module) |
| Background monitor | `EndpointMonitoring.MonitoringService` | Windows Service |

Both share a single SQLite database; in addition, the monitoring service pushes real-time check results to the web app's SignalR hub (see section 5). The `.NET Aspire AppHost` project is for local development only and is **not deployed**.

---

## Prerequisites

### On the Windows Server

1. **IIS** — enable via *Server Manager → Add Roles and Features → Web Server (IIS)*  
   Required IIS features: Static Content, Default Document, Directory Browsing, HTTP Errors, HTTP Redirection, ASP.NET, **WebSocket Protocol** (under *Web Server → Application Development* — required for Blazor Server and the SignalR real-time updates)

2. **[.NET 10 Hosting Bundle](https://dotnet.microsoft.com/download/dotnet/10.0)**  
   Installs the .NET Runtime, ASP.NET Core Runtime, and the IIS ASP.NET Core Module (ANCM).  
   Restart IIS after installation: `iisreset`

3. **Firewall** — open TCP 80 / 443 as needed.

---

## 1. Build and Publish

Run these commands on your development machine from the repository root.

```powershell
# Web UI
dotnet publish src\EndpointMonitoring.Web `
  -c Release -o publish\web

# Background service
dotnet publish src\EndpointMonitoring.MonitoringService `
  -c Release -o publish\service
```

Then copy both output folders to the server, for example:

- `C:\inetpub\wwwroot\endpoint-monitoring\` ← contents of `publish\web`
- `C:\Services\endpoint-monitoring\` ← contents of `publish\service`

### Versioning

Every Release build is stamped automatically with a CalVer version (see
`src/Directory.Build.props`): `YYYY.R.MMdd.HHmm`, e.g. `2026.1.0606.1432`
= release 1, built June 6 2026 at 14:32. The short git commit hash is
appended to the informational version (`2026.1.0606.1432+ab12cd3`).

- Bump `<ReleaseNumber>` in `src/Directory.Build.props` for a new release;
  date/time and commit hash are filled in at build time.
- The running version is shown at the bottom of the navigation drawer in
  the Web UI and logged at startup by both the web app and the service.
- To check a deployed binary: file properties → Details → *Product version*.

## 2. Deploy the Web App to IIS

1. Open **IIS Manager**.

2. Right-click **Sites → Add Website**:
   - **Site name**: `EndpointMonitoring`
   - **Physical path**: `C:\inetpub\wwwroot\endpoint-monitoring`
   - **Binding**: choose HTTP (port 80) or HTTPS (port 443)

3. Open the **Application Pool** created for the site:
   - Set **.NET CLR version** to **No Managed Code**
   - Set **Identity** to an account with write access to the database folder  
     (or use *ApplicationPoolIdentity* and grant it permissions on the DB folder)

4. The `web.config` generated by `dotnet publish` wires up the ASP.NET Core Module automatically — no edits needed.

> **HTTPS note:** In the Production environment, the app redirects all HTTP traffic to HTTPS. Either add an HTTPS binding with a certificate in IIS, or temporarily set the environment variable `ASPNETCORE_ENVIRONMENT=Development` on the App Pool to skip the redirect while testing.

---

## 3. Deploy the Monitoring Service

Run these commands **on the server** in an elevated PowerShell session:

```powershell
sc.exe create "EndpointMonitoringService" `
  binPath="C:\Services\endpoint-monitoring\EndpointMonitoring.MonitoringService.exe" `
  start=auto `
  displayname="Endpoint Monitoring Service"

sc.exe start EndpointMonitoringService

# Verify
sc.exe query EndpointMonitoringService
```

To remove the service later:

```powershell
sc.exe stop EndpointMonitoringService
sc.exe delete EndpointMonitoringService
```

## 4. Database Configuration

Both services default to:

```
C:\ProgramData\EndpointMonitoring\monitoring.db
```

The folder and database file are created automatically on first start. Both services must point to the **same path**.

To use a custom location, add (or update) `appsettings.json` in both deployment folders:

```json
{
  "DatabasePath": "D:\\Data\\EndpointMonitoring\\monitoring.db"
}
```

Or set it as an environment variable on each process:

| Process | Where to set |
|---|---|
| Web (IIS) | App Pool → Advanced Settings → Environment Variables |
| Windows Service | System environment variables or registry `HKLM\SYSTEM\CurrentControlSet\Services\EndpointMonitoringService\Environment` |

**Permissions:** The IIS App Pool identity and the Windows Service account must both have **read/write access** to the database folder.

```powershell
# Example: grant IIS AppPool access (replace "EndpointMonitoring" with your pool name)
icacls "C:\ProgramData\EndpointMonitoring" /grant "IIS AppPool\EndpointMonitoring:(OI)(CI)F"
```

## 5. Email Alerts & Real-Time Updates

### SMTP (email alerts)

Configure the `Smtp` section in `appsettings.json` in **both** deployment folders — the monitoring service sends the alert emails, the web app uses it for the *Send Test Email* feature:

```json
{
  "Smtp": {
    "Host": "smtp.example.com",
    "Port": 587,
    "UseSsl": false,
    "Username": "",
    "Password": "",
    "FromAddress": "monitoring@example.com",
    "FromName": "Endpoint Monitoring"
  }
}
```

Port 587 uses STARTTLS (default); for implicit SSL on port 465 set `UseSsl: true`. Username/password are optional. Leave `Host` empty to disable email alerts.

### `WebsiteUrl` (monitoring service only)

Set `WebsiteUrl` in the **monitoring service's** `appsettings.json` to the public URL of the web app:

```json
{
  "WebsiteUrl": "https://monitoring.example.com"
}
```

It is used for two things:

1. **Alert emails** — included as a link back to the dashboard.
2. **Real-time dashboard updates** — the service connects to the web app's SignalR hub at `<WebsiteUrl>/hubs/monitoring` and pushes each check result, so dashboard rows update live without a page refresh.

If `WebsiteUrl` is missing, monitoring and email alerts still work — only the real-time push is disabled (the service logs a warning at startup). The hub endpoint allows anonymous access and the client accepts self-signed certificates, so internal HTTPS setups work without extra configuration. If the web app is temporarily unreachable, the service retries every 30 seconds and reconnects automatically.

## 6. Authentication Configuration

### Local users (default)

The app seeds a default admin account on first start using the value in `appsettings.json`:

```json
{
  "Auth": {
    "DefaultAdmin": {
      "Username": "admin",
      "Password": "ChangeMe!123"
    }
  }
}
```

**Change the default password before or immediately after first login.**

### Optional: OpenID Connect (Zitadel or any OIDC provider)

#### Step 1 — Register the application in your OIDC provider

Create a new application (type: **Web**, grant type: **Authorization Code + PKCE**) and register these exact URIs — replace `https://monitoring.example.com` with your actual host:

| Setting in provider | Value |
|---|---|
| **Redirect URI** (login callback) | `https://monitoring.example.com/signin-oidc` |
| **Post-logout redirect URI** | `https://monitoring.example.com/signout-callback-oidc` |

If you need to test over plain HTTP first, also add the `http://` variants — most providers allow multiple URIs per application.

> These paths are fixed in the application and cannot be changed via configuration.

#### Step 2 — Configure `appsettings.json`

Update `appsettings.json` in the **Web** deployment folder with the values from your provider:

```json
{
  "Oidc": {
    "Enabled": true,
    "Authority": "https://<instance>.zitadel.cloud",
    "ClientId": "<client-id>",
    "ClientSecret": "<client-secret>",
    "Scopes": ["openid", "profile", "email", "urn:zitadel:iam:org:project:roles"],
    "RolesClaim": "urn:zitadel:iam:org:project:roles",
    "AdminRoleValue": "admin",
    "ViewerRoleValue": "viewer"
  }
}
```

| Field | Description |
|---|---|
| `Authority` | Base URL of your OIDC provider (the issuer) |
| `ClientId` | Application/client ID from the provider |
| `ClientSecret` | Client secret from the provider |
| `Scopes` | Scopes to request — defaults work for Zitadel |
| `RolesClaim` | JWT claim that contains the user's role(s) |
| `AdminRoleValue` | Value within that claim that maps to the Admin role |
| `ViewerRoleValue` | Value within that claim that maps to the Viewer role |

When OIDC is enabled, external users are automatically provisioned in the local database on first login. Local user accounts continue to work alongside OIDC accounts.

## 7. Optional: OpenTelemetry

Both services export telemetry via OTLP if the endpoint is configured. Set the environment variable on each process:

```
OTEL_EXPORTER_OTLP_ENDPOINT=http://your-collector:4317
```

Leave it unset to disable telemetry export entirely.

## 8. Verify the Deployment

```powershell
# Windows Service is running
sc.exe query EndpointMonitoringService

# Web app responds (adjust host/port)
Invoke-WebRequest http://localhost -UseBasicParsing
```

Open `http://<server>/Account/Login` in a browser — you should see the login page.  
Log in with the admin credentials and confirm the dashboard loads and endpoints are being monitored.

**Real-time updates:** after the next check cycle, a green chip with the last-received-signal timestamp appears in the dashboard's *Endpoint Status* toolbar and updated rows flash briefly. If it never appears, check the monitoring service log for `No hub URL configured` (missing `WebsiteUrl`) or hub connection warnings, and verify the IIS **WebSocket Protocol** feature is installed.

**Email alerts:** on the *Users* page, use the per-user **Send Test Email** button to verify SMTP connectivity.

## Updating to a New Version

1. Publish the updated project(s) on your dev machine (same commands as above).
2. **Web:** stop the IIS site, overwrite the deployment folder, restart the site.
3. **Service:** stop the service, overwrite the deployment folder, start the service.

```powershell
# Service update
sc.exe stop EndpointMonitoringService
# ... copy new files ...
sc.exe start EndpointMonitoringService
```

The database schema is managed via `EnsureCreated` — no manual migration step is required for additive changes.
